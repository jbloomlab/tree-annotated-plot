## Workflow shape

The realistic use is **two-stage**:

1. **Stage 1 (somewhere else, possibly long before)** — the user runs their
   analysis pipeline and saves a titer chart:
   ```python
   chart = alt.Chart(...)...
   chart.save("titers.json")   # Vega-Lite JSON
   ```
2. **Stage 2 (us)** — the user invokes our CLI or function against the saved
   file plus the tree:
   ```
   tree-annotated-plot \
       --chart-spec titers.json \
       --tree h3n2.auspice.json \
       --output combined.html
   ```

The in-process use (`tap.plot(chart=alt.Chart(...), tree=...)` from a single
notebook) still works and is supported. But it is not the assumed default.

## Chart input formats

JSON is canonical; HTML is supported as a best-effort convenience for users
who only have a saved page lying around.

Both the Python function and the CLI accept the same set of chart inputs,
loaded by a shared `_load_chart` helper:

- **`alt.Chart` (or `LayerChart` / `FacetChart` / `HConcatChart` /
  `VConcatChart` / `ConcatChart`)** — in-process use. Used as-is.
- **Path to a Vega-Lite JSON file** (`*.json`) — saved via `chart.save("...")`.
  Loaded with `alt.Chart.from_dict(json.load(...))`. Canonical CLI input.
- **`dict`** — already-parsed Vega-Lite spec. Same path through `from_dict`.
- **Path to an HTML file** (`*.html`) — saved via `chart.save("foo.html")`.

#### Data sourcing inside the spec — inline and named only; refuse URL

Vega-Lite specs reference data three ways:

- **Inline** via `data.values` — the data is a list literal in the spec.
- **Named** via `data.name` plus a top-level `datasets[name]` block.
- **URL** via `data.url` — the data is fetched from disk or HTTP at render
  time.

We support **inline** and **named**. We **refuse URL data** with a clear
`ValueError`: the package would have to fetch the data to extract chart
strain values for tip-set reconciliation, and we don't want to do
synchronous I/O (with the attendant network failures, auth surprises, and
"the file moved" mysteries) at plot time. Saved altair charts default to
inline, so this matters only for hand-crafted specs. The error message
points the user at materializing the data inline before saving.
  We extract the embedded Vega-Lite spec from the page using a small
  stdlib-only helper (see below). Documented as best-effort: works on
  charts saved with the default altair template, may fail on custom
  `chart.save(template=...)` outputs.

### HTML extraction approach (no regex, no new dependencies)

altair's default `to_html()` template emits a script block of the form:
```js
(function(vegaEmbed) {
  var spec = {...};   // valid JSON, produced by json.dumps
  ...
  vegaEmbed("#vis", spec, embedOpt).catch(...);
})(vegaEmbed);
```

We extract the spec using two stdlib pieces:

- **`html.parser.HTMLParser`** — collects the text content of every
  `<script>` tag. Handles real HTML structure, comment blocks, and so on
  without us having to think about it.
- **`json.JSONDecoder.raw_decode(s, idx)`** — parses a single JSON value
  starting at index `idx` in `s`, returning `(value, end_idx)`. Does
  brace-balancing correctly, including string-internal braces and escape
  sequences. This is the right tool for "find one JSON object embedded in
  a larger string".

Implementation sketch:

```python
from html.parser import HTMLParser
import json


class _ScriptCollector(HTMLParser):
    def __init__(self):
        super().__init__()
        self.scripts: list[str] = []
        self._in_script = False

    def handle_starttag(self, tag, attrs):
        if tag == "script":
            self._in_script = True

    def handle_endtag(self, tag):
        if tag == "script":
            self._in_script = False

    def handle_data(self, data):
        if self._in_script:
            self.scripts.append(data)


def _extract_spec_from_html(html: str) -> dict:
    parser = _ScriptCollector()
    parser.feed(html)
    decoder = json.JSONDecoder()
    for script in parser.scripts:
        i = script.find("var spec")
        if i == -1:
            continue
        brace = script.index("{", i)
        spec, _ = decoder.raw_decode(script, brace)
        return spec
    raise ValueError(
        "no `var spec = {...}` block found in HTML; this is likely a "
        "non-default altair template. Re-save as JSON: "
        "chart.save('foo.json')."
    )
```

About 25 lines, zero regex, zero third-party deps. The error message points
the user at the canonical JSON path when extraction fails.

**Limits, documented up front:**

- Works on the default altair template. `chart.save(template=...)` outputs
  may not match `var spec = {...}` and will raise `ValueError` with a
  remediation message.
- We do not parse JS in general. If altair changes its template in a
  future release (e.g., to `let spec`, `const spec`, or inlining the spec
  into the `vegaEmbed(...)` call directly), extraction breaks until we
  update the locator. This is captured in the version-compatibility check
  alongside the Vega-Lite schema check.
- A page with multiple `var spec = {...}` blocks: we take the first.
  Document this as a known limit; multi-chart HTML is unusual.

## Real test data — what the saved charts actually look like

The user's `examples/flu-seqneut-2025to2026_titer_charts.py` produces two
saved charts. Their structure is *not* a flat `alt.Chart` and the plan must
handle this correctly:

- **Top level**: `alt.vconcat(faceted, dummy_cohort, spacing=1)` with a chain
  of `.configure_axis(...)`, `.configure_header(...)`, `.configure_view(...)`,
  `.configure_facet(...)`, `.configure_legend(...)`, `.properties(title=...)`.
  → a `VConcatChart` at the root.
- **`faceted`**: `body.facet({facet_dim: alt.Column("cohort_n:N", ...)})`
  → a `FacetChart` whose `.spec` is `body`.
- **`body`**: `layer + median_points` → a `LayerChart` with two layers; both
  layers share the strain encoding on `axis_label:N`.
- **`dummy_cohort`**: a separate small `Chart` with no strain encoding (it's
  the legend-binding hack).

That means chart introspection has to:
1. Walk into `VConcatChart`, `HConcatChart`, `ConcatChart`, `FacetChart`,
   `LayerChart`.
2. Find every encoding that references `chart_strain_field`.
3. Verify they all use the same axis (`x` or `y`) and the same value-set.
4. Tolerate panels that *don't* have the strain encoding (the dummy_cohort
   legend, axis labels, etc.).

Two orientations are present in the real data:

- **H3N2 (`chart_type="iqr"`)**: `vertical=True` → strain on `y`,
  `height=alt.Step(11)`, `width=FACET_SIZE=150`. → vertical strain layout,
  tree on the **left**, hconcat.
- **H1N1 (`chart_type="lines"`)**: `vertical=False` → strain on `x`,
  `width=alt.Step(11)`, `height=FACET_SIZE=150`. → horizontal strain layout,
  tree on **top**, vconcat.

Both charts encode the strain axis on `"axis_label:N"` (the
`derived_haplotype`-with-fallback column), **not** on `"virus"`. The original
strain identifier in the data is `"virus"`. This is the central case for the
"chart strain field is not the same as tree strain field" feature: the
chart-side join key is `axis_label` while the tree-side tip name is some
attribute of an Auspice node.

The Auspice JSONs that go with these are downloaded from:

- H3N2: <https://github.com/jbloomlab/flu-seqneut-2025to2026/blob/main/auspice/flu-seqneut-2025to2026_H3N2.json>
- H1N1: <https://github.com/jbloomlab/flu-seqneut-2025to2026/blob/main/auspice/flu-seqneut-2025to2026_H1N1.json>

The join attribute on each tip is **`derived_haplotype`** (under
`node_attrs`), matching the chart's `axis_label` column. So
`tree_strain_field="derived_haplotype"` and
`chart_strain_field="axis_label"` for both example invocations.

**Open question for the user:** which Auspice node attribute on these trees
holds the value that matches the chart's `axis_label` column? The tip's
`name` field will likely be the full virus identifier
(e.g., `A/Sydney/1359/2024_H3N2`) which **does not equal** the
`axis_label` (`derived_haplotype` if present, else virus name with the
`_H3N2`/`_H1N1` suffix stripped). Three ways to resolve:

1. **Tree side adds an attribute** — the Auspice JSON exposes
   `node_attrs.derived_haplotype` (or similar) on each tip; we use that as
   `tree_strain_field` and the chart's `axis_label` matches.
2. **Chart side encodes the join key** — change the chart-builder script to
   encode `virus:N` on the strain axis (with `axis_label` only as a `labelExpr`
   or tooltip), so that join-on-`virus` works directly.
3. **Package supports a join-mapping argument** — explicitly take a callable
   or dict that maps tree-tip values → chart-strain values. More flexible but
   more API surface.

I recommend **(1) or (2) over (3)**: the package's job is to align two
columns by equality, not to do data wrangling. (3) creeps the package into
ETL territory. The user should pick (1) or (2) before Phase 2a starts; the
choice may also depend on whether the user wants to control axis labels
independently of the join key.

## Strain identity and ordering — the headline feature

This is the central concern of Phase 2 and deserves its own section.

### The reorder-chart-to-match-tree rule

The package's job: **the chart's strain axis is re-sorted to the order the
tree gives.** Whatever order the chart originally had — whether
(subclade, virus) like the Kikawa scripts, or alphabetical, or unsorted — is
overridden. The tree's tip-traversal order is authoritative.

This was already happening in Phase 1 (the y-sort override via deepcopy), but
the previous plan treated it as plumbing. Phase 2 promotes it to the
documented headline behavior:

- The README leads with "tree dictates strain order; chart follows".
- The `plot()` docstring leads with the same sentence.
- The internal helper that does the override gets a name and a docstring
  that makes this explicit (e.g., `_apply_tree_order_to_chart_strain_axis`).

### `chart_strain_field` and `tree_strain_field`

Two separate parameters with two separate roles. **Both are required; neither
has a default.** Forcing the user to spell out both names removes any
ambiguity about which side names what.

- **`chart_strain_field`** — the *data-column name* the chart's strain axis
  encodes. In the Kikawa H3N2 example this is `"axis_label"`; in the H1N1
  example it is also `"axis_label"`. This is the field name as it appears
  in the chart's `data` (or upstream of any `transform_lookup`/`transform_calculate`
  chain) and in the encoding's `field` property. Auto-detection is fragile
  when the chart has multiple categorical encodings (`color=cohort`,
  `detail=serum`, etc.), so we require the user to be explicit.
- **`tree_strain_field`** — the *Auspice node attribute* holding the value
  that matches `chart_strain_field`. Lookup rules are intentionally simple
  and **non-dotted**:
  - The literal value `"name"` means **the top-level node `name` field** in
    the Auspice JSON (the canonical tip identifier in nextstrain Auspice v2,
    e.g., `"A/Sydney/1359/2024_H3N2"`).
  - Any other value `X` means `node_attrs[X]` — a key directly under
    `node_attrs`. We do **not** accept dotted paths
    (`"node_attrs.derived_haplotype"`); just pass `"derived_haplotype"`.
  - When `node_attrs[X]` is itself a dict with a `"value"` key (the Auspice
    convention used by `num_date`, `clade_membership`, `derived_haplotype`,
    etc.), we automatically unwrap to that `value`. So
    `tree_strain_field="derived_haplotype"` resolves to
    `node["node_attrs"]["derived_haplotype"]["value"]`.
  - Anything else (a string lookup that misses, a non-dict-non-scalar
    value) → `ValueError`.

  This covers the two real-world cases (top-level `name`, or a tagged
  `node_attrs.X.value` per Auspice convention) without exposing path-syntax
  complexity. If a future case needs deeper paths, we revisit then.

  *Note about Auspice naming*: Auspice's standard tip identifier sits at the
  **top-level `name` field on each node**, not under `node_attrs`. There is
  generally no `node_attrs.strain` or `node_attrs.name`. So `"name"` is the
  one magic value, and everything else lives in `node_attrs`.

The match is by **value equality**: for each chart-data row, the value of
`chart_strain_field` must equal the value at `tree_strain_field` on exactly
one tree tip. This is the primary cross-package join.

`TreeNode.name` (currently the tip's top-level `name` field in Phase 1)
becomes "the value at `tree_strain_field`". Internal Auspice node IDs and
user-facing strain names get cleanly separated.

### Axis-row uniqueness — one axis row per distinct chart strain

A common subtle failure mode: the chart data has many rows per strain
(e.g., one per `(serum, virus)` pair), and the user's encoding collapses
them onto axis rows. We must enforce that **each value of
`chart_strain_field` corresponds to exactly one axis row**, even though it
can appear in many data rows. Two cheap checks do this reliably:

1. **The strain encoding's `type` must be `"nominal"` or `"ordinal"`.**
   Vega-Lite categorical encodings already collapse duplicate data values
   onto a single axis row; a `"quantitative"` or `"temporal"` encoding
   would not. Reject anything other than `nominal`/`ordinal` with a
   `ValueError`.
2. **The strain encoding must be on `x` or `y`.** A `chart_strain_field`
   bound to `color`, `detail`, `tooltip`, etc. cannot be tree-aligned. We
   detect this and raise (see "`chart_strain_field` axis check" below).

Together these guarantee one axis row per distinct value, regardless of
how many data rows feed it.

### Tip-set reconciliation: prune / error / warn

After loading the tree and finding the chart's strain values, we have two
sets:

- `tree_strains` — the set of tip values at `tree_strain_field`.
- `chart_strains` — the set of distinct chart-data values at
  `chart_strain_field`.

There are three possible asymmetries; each gets a controlled policy:

1. **Chart strains not in the tree** (`chart_strains - tree_strains`).
   Default: **fatal `ValueError`**. Rationale: dropping these would silently
   lose data the user explicitly chose to plot. There is **no opt-out** for
   this case.
2. **Tree tips not in the chart** (`tree_strains - chart_strains`). Default:
   **fatal `ValueError`**. Opt-in to **drop them** with
   `prune_tree_to_chart=True` (Python) / `--prune-tree-to-chart` (CLI).
   When pruned, the tree is reduced to the subtree spanning only chart
   strains, with internal nodes whose subtrees become empty removed;
   segments and layout are recomputed on the pruned tree. This is in-memory
   only — the on-disk Auspice JSON is untouched.
3. **Duplicate `tree_strain_field` values across tips** on the tree side.
   Fatal `ValueError` listing the duplicates. The chart side is *not*
   subject to this check: many chart-data rows can share the same
   `chart_strain_field` value (that is normal — the categorical encoding
   collapses them onto one axis row, validated separately above).

#### Error messages must help the user diagnose wrong-field mistakes

The most likely cause of a tip-set mismatch is **the user passed the wrong
field name** (chart-side, tree-side, or both) — not that the data is
genuinely incomplete. So both error paths above include three things in
the message, in this order:

1. **What was tried**, named explicitly: `chart_strain_field=<X>`,
   `tree_strain_field=<Y>`.
2. **Sample values from each side** — up to 5 distinct values from
   `chart_strains` and 5 from `tree_strains`, sorted, so the user can
   eyeball whether the values look like the same kind of identifier
   (e.g., are both sets formatted like `A/Sydney/1359/2024_H3N2`, or is
   one side a clade label and the other a virus name?).
3. **Candidate fields the user may have meant**, computed by:
   - On the *chart* side: enumerate every encoding `field` that appears
     anywhere in the spec at a categorical type. For each, compute the
     overlap between its distinct data values and `tree_strains`. Any
     field with overlap ≥ 50% (tunable) gets listed: `"the chart has a
     field 'virus' whose values match 28 / 30 tree strains — did you mean
     chart_strain_field='virus'?"`.
   - On the *tree* side: enumerate every key under `node_attrs` (across
     all tips, including those with the `{"value": ...}` shape unwrapped),
     plus the top-level `"name"` field. For each, compute the overlap
     between its tip-value set and `chart_strains`. Same threshold and
     report format: `"the tree has node_attrs['derived_haplotype'] whose
     values match 25 / 28 chart strains — did you mean
     tree_strain_field='derived_haplotype'?"`.

This is a one-pass scan over the chart spec and the tree node list; cheap
and runs only on the error path.

Example diagnostic:

```
ValueError: 28 chart strains are not present in the tree.

Tried: chart_strain_field='axis_label', tree_strain_field='name'
Sample chart_strain_field values: ['K:K207Q,V223I', 'K:Q80K', 'K:S96C,K207Q,V223I', ...]
Sample tree_strain_field values:  ['A/Sydney/1359/2024_H3N2', 'A/Wisconsin/67/2022_H3N2', ...]

These look like different kinds of identifier. Possible alternatives:
  - The chart has a field 'virus' whose values match 28 / 28 tree strains
    — did you mean chart_strain_field='virus'?
  - The tree has node_attrs['derived_haplotype'] whose values match
    27 / 28 chart strains — did you mean
    tree_strain_field='derived_haplotype'?
```

Pruning details (the **strict** flavor, locked):

- The pruning preserves the topology among kept tips: every internal node
  with ≥1 kept descendant stays; nodes whose entire subtree is dropped are
  removed.
- An internal node left with a single child after pruning is **collapsed**
  into its child. The child keeps its identity; the internal node is
  discarded.
  - **`branch_length="div"`**: the dropped internal's `div` delta is
    *summed* into the surviving child's effective branch length, so the
    distance from the kept root to each tip is preserved. (Practically:
    we recompute branch lengths from `parent.div - child.div` after the
    collapse, or pre-store the child's absolute `div` and let the layout
    code subtract from the new parent.)
  - **`branch_length="num_date"`**: dates are absolute *positions*, not
    deltas. The collapsed internal's date is simply discarded; the
    surviving child keeps its own `num_date` value. No summation.
  - The collapsed internal's `node_attrs` are discarded entirely. The
    child's `node_attrs` are kept as-is. (We don't try to merge metadata
    — there's no meaningful merge for two clade memberships, two haplotype
    labels, etc.)
- After pruning + single-child collapse, **re-root the tree to the LCA**
  (lowest common ancestor) of the kept tips. If all kept tips live in one
  subtree of the original root, the new root is deeper than the old root,
  and we don't draw a long meaningless stem from the original root.
- Branch lengths (`div` or `num_date.value`) on kept edges are otherwise
  preserved. Nothing is rescaled.

### Tree tip ordering after pruning

Tip order is determined by depth-first traversal of the post-prune tree. The
existing `_tree.tips()` function gives this; it just runs on the pruned
copy.

## Version-compatibility checks on inputs

(Carried forward from the previous plan; same design, summarized.)

### Chart spec (Vega-Lite / altair)

At chart-load time, inspect `$schema`. If the URL identifies Vega-Lite v5 or
earlier:

- Default: raise `ValueError` with a remediation message (*"this chart spec
  was saved with Vega-Lite 5 (likely altair 5); please re-save it from an
  altair 6+ environment with `chart.save(...)`"*).
- With `strict_version=False`: downgrade to `warnings.warn(...)`.
- Missing `$schema`: emit a warning and proceed.

The chart check only applies when the chart input went through `from_dict`
(JSON / dict). It has no effect on a live `alt.Chart`.

### Tree JSON (Auspice)

At tree-load time, inspect the top-level `version`:

- If present and not starting with `"v2"`: raise `ValueError` (or warn under
  `strict_version=False`).
- If absent: warn and proceed.

### Strict version is only an advisory layer

The version checks are an early-fail guard for the most common cross-version
mishaps. They are **not** sufficient validation. The parsers must remain
strict about required fields *regardless* of `strict_version`:

- Missing `node_attrs` on a tree node → `ValueError`.
- Missing `chart_strain_field` encoding in the chart → `ValueError`.
- `branch_length="num_date"` but no `node_attrs.num_date.value` → `ValueError`.
- No silent imputation of required fields, ever.

Optional metadata fields (chart titles, unrelated node attrs) may be missing
without comment.

## Two API surfaces, one config

Both surfaces are thin wrappers over a single
`_build(chart, tree, config) -> alt.Chart` implementation. The shared spec
lives as a `PlotConfig` dataclass. Adding a new parameter means: add a field
to `PlotConfig`, expose it as a function kwarg and a CLI flag, document it
once in the dataclass docstring.

### Python function

```python

    chart,                            # alt.Chart | LayerChart | FacetChart

                                      #   | str | Path     (Vega-Lite JSON or HTML)
                                      #   | dict           (already-parsed spec)

    *,
    chart_strain_field,               # required: data-column name in the chart

either agree (no-op) or disagree (we would then refuse to render because

```

## Parameter design

### `chart` and `chart_strain_field`

The chart can be any of `Chart`, `LayerChart`, `FacetChart`, `HConcatChart`,
`VConcatChart`, `ConcatChart`, **or any of those serialized to a JSON file
or a saved altair HTML page**. After loading, the package walks the chart
spec to find every encoding that references `chart_strain_field`.

**Required; no default.** Charts often have multiple categorical encodings
(e.g. `y=axis_label`, `color=cohort`); auto-detection would be fragile.

#### `chart_strain_field` axis check

After locating every encoding that references `chart_strain_field` (across
every layer, every facet panel, and every concat sub-chart), partition the
hits into two groups:

- **Axis hits** — encodings on the `x` or `y` channel. These determine the
  tree alignment.
- **Secondary hits** — encodings on any other channel (`color`, `tooltip`,
  `detail`, `size`, `opacity`, `shape`, `order`, `text`, etc.). These are
  unrelated to alignment and pass through untouched.

Validation rules apply only to the axis hits:

1. **At least one axis hit exists.** Zero axis hits → `ValueError` with
   the candidate-fields hint described under "Tip-set reconciliation".
   (If `chart_strain_field` is *only* on, say, a tooltip and never on x
   or y, the chart isn't aligning anything to a tree.)
2. **All axis hits agree on the same channel** — every axis hit is on `x`,
   or every axis hit is on `y`. Mixing (one layer on `x`, another on `y`,
   or even the same panel encoding it on both) → `ValueError` because the
   target axis is ambiguous.
3. **Each axis hit's encoding type is `nominal` or `ordinal`.** A
   `quantitative` or `temporal` strain encoding wouldn't collapse duplicate
   data values onto a single axis row; reject it. (See "Axis-row
   uniqueness" above.)

**Secondary hits on `chart_strain_field` are explicitly allowed and not
checked.** Real charts routinely surface the strain identifier in
tooltips, color encodings, or `detail` channels in addition to the axis
encoding — that's common and good practice. The package does not touch
those hits, doesn't require them to use the same data values as the axis
hit (they could, e.g., be a transformed-lookup version), and ignores them
for the purposes of tree alignment.

Example error messages:

- Zero axis hits, but a tooltip hit exists → `ValueError`:
  `"chart_strain_field='axis_label' is encoded on 'tooltip' but not on 'x' or 'y'; tree alignment requires it on an axis channel."`
- One layer puts it on `x`, another on `y` → `ValueError`:
  `"chart_strain_field='axis_label' is encoded on both 'x' (in panel … layer …) and 'y' (in panel … layer …); the alignment axis is ambiguous."`

### `tree` and `tree_strain_field`

`tree_strain_field` is **required, no default**. Lookup rules (repeating
from the strain-identity section so this stays self-contained):

- Literal `"name"` → top-level node `name` field.
- Anything else → `node_attrs[<value>]`, with auto-unwrap of the Auspice
  `{"value": ...}` convention.
- No dotted paths.

The current Phase-1 `TreeNode.name` becomes "the resolved value at
`tree_strain_field`". Internal tree IDs vs. user-facing strain names get
cleanly separated.

### `branch_length`

`"div"` (default) or `"num_date"`. Auspice `node_attrs.div` is a scalar;
Auspice `node_attrs.num_date` is `{"value": 2020.5, "confidence": [...]}`.
The scalar `value` field is what we want.

When `num_date`, the tree's branch axis runs from oldest (root) to most
recent (tips). When `div`, root at div=0 (or whatever the root's value is).

The scale bar (below) needs to know which mode it's in to label units.

### `tree_size` and tip-axis alignment with the chart

**Decided:** fraction default. (User answer 2.)

`tree_size` controls only the tree's **branch-axis** dimension (the
direction perpendicular to the strain rows). Accepts:

- `int` ≥ 1 → absolute pixels along the tree's branch axis.
- `float` in (0, 1] → fraction of the chart's strain-axis dimension.

Default **`0.3`** — empirically reasonable.

The tree's **tip-axis** dimension (the direction along the strain rows) is
*not* configurable: it is computed to match the chart's strain-axis
dimension exactly so each tree tip aligns with its row in the chart. We
detect the chart's sizing strategy from the chart's spec:

- **`alt.Step(N)` sizing** (e.g., the H3N2 chart sets
  `height=alt.Step(11)`). The chart's strain-axis size is `N × n_strains`
  in pixels. We detect the `Step(N)` value by inspecting the relevant
  `width`/`height` property in the (possibly nested) chart spec, and
  propagate the same `alt.Step(N)` to the tree's tip axis. Vega-Lite then
  sizes both at the same per-row pitch so rows align row-for-row.
- **Fixed pixel sizing** (e.g., `width=600`, `height=400`). The chart's
  strain-axis size is the fixed value. We use the same value as the
  tree's tip-axis size.
- **Anything else** (continuous-axis sizing, missing dimension): raise
  `ValueError` pointing the user at `Step(N)` or a fixed pixel value as
  the supported sizing strategies for the strain axis.

For concat / facet charts, "the chart's strain-axis dimension" means the
dimension on the panel that carries the `chart_strain_field` encoding —
the same panel found by the axis-check walk.

### `tree_line_width`, `tree_node_size`, `leader_line_width`

Direct passthrough to the layered tree chart (currently hardcoded 1.5, 28,
1.0 in Phase 1).

- `tree_line_width` — branch stroke width.
- `tree_node_size` — area of the tip-circle marks. **`tree_node_size=0`
  disables the tree-tip circle layer entirely** (use this when you want
  unmarked branches that just terminate at leader lines, or for trees so
  dense the dots clutter the figure).
- `leader_line_width` — width of the dashed leader lines from each tip to
  the chart row. **`leader_line_width=0` (or `None`) disables the
  leader-line layer entirely** — useful when the rightmost tip-spread is
  small enough that leaders aren't needed.

### `scale_bar` and `branch_length_units`

**`scale_bar`** is a boolean, off by default. When True, draw a small
horizontal scale bar with a label below the tree panel (and inside the
tree's allotted space). Length is auto-computed:

- Total tree branch-axis range = `b_max - b_min`.
- Scale bar length = the largest "nice" round number ≤ 25% of that range
  (powers of 10, or 1/2/5 × power of 10).

**`branch_length_units`** is the user-facing unit string we paste after the
numeric length on the scale-bar label. **It must be specified by the user**
(or left `None` for unitless display) when `scale_bar=True` and
`branch_length="div"`. We **cannot reliably auto-detect** divergence units
from an Auspice JSON because:

- Augur exports may emit `node_attrs.div` in either *substitutions per
  site* (a rate, typical when an alignment-based clock model is used) or
  *substitutions* (a count, when raw substitution counts are exported).
- Auspice JSON v2 has no formal field for the units; the meta dict
  *sometimes* hints (e.g., a `meta.extensions.nextstrain.metadata` block)
  but it is not standardized and many exports omit it altogether.
- A magnitude heuristic (small → rate, large → count) misclassifies any
  tree whose values straddle the threshold and is silently wrong on
  those.

So the policy is: ask the user. Common values: `"substitutions/site"`,
`"substitutions"`, `"amino-acid changes"`, etc. For `branch_length="num_date"`
we ignore `branch_length_units` and label automatically (e.g. `"5 years"`,
`"6 months"`).

Concrete labels:

- `branch_length="div"`, `branch_length_units="substitutions/site"`,
  bar length 0.01 → label `"0.01 substitutions/site"`.
- `branch_length="div"`, `branch_length_units=None`, bar length 0.01 →
  label `"0.01"`.
- `branch_length="num_date"`, bar length 5.0 → label `"5 years"`.
- `branch_length="num_date"`, bar length 0.5 → label `"6 months"`.

Implementation: a 4th layer in the tree chart (line + text), positioned below
the tip rows.

### Orientation: derived from `chart_strain_field`'s axis (no parameter)

There is no `orientation` parameter (see the function signature above for
why). Auto-detection is now the *only* path:

- Strain on `y` → **vertical** layout: tree on the **left**, root at left,
  branches grow rightward, tips on the right of the tree, `hconcat` with
  chart. (H3N2 case.)
- Strain on `x` → **horizontal** layout: tree on **top**, root at top,
  branches grow downward, tips at the bottom of the tree, `vconcat` with
  chart. (H1N1 case.)

Tree drawing is reused across layouts via a single coordinate system where
the "branch axis" and "tip axis" are abstract; the renderer binds them to
`x`/`y` based on the detected layout.

### `prune_tree_to_chart`

Boolean, default `False`. See "Tip-set reconciliation" above.

### `strict_version`

Boolean, default `True`. Single flag controlling both the chart-version
check (Vega-Lite schema URL) and the tree-version check (Auspice top-level
`version` field). See "Version-compatibility checks" above.

## Documentation generation

**Decided plan:** **MkDocs Material + mkdocstrings + mkdocs-click**, with
**`PlotConfig` as the single source for parameter descriptions** —
including the per-parameter help text that appears in *both* the function
docstring rendering and the CLI `--help` output.

### One source of parameter docs for both surfaces

The naive setup duplicates each parameter's description in two places: the
function docstring's `Args:` section, and each click `@click.option(...,
help="...")`. We avoid that.

`PlotConfig` is a dataclass whose fields use `typing.Annotated` to attach a
description to each parameter:

```python
from dataclasses import dataclass
from typing import Annotated, Literal

@dataclass(frozen=True, kw_only=True)
class PlotConfig:
    chart_strain_field: Annotated[
        str,
        "Name of the data column in the chart that holds strain identifiers. "
        "Must be encoded on the chart's x or y axis as a nominal/ordinal field.",
    ]
    tree_strain_field: Annotated[
        str,
        "Where to find the matching strain identifier on each tree tip. "
        "Use 'name' for the top-level Auspice node `name` field; otherwise "
        "the value of `node_attrs[<this>]` (with the `{value: ...}` Auspice "
        "convention auto-unwrapped).",
    ]
    branch_length: Annotated[
        Literal["div", "num_date"],
        "Which Auspice node attribute supplies branch lengths. 'div' uses "
        "node_attrs.div; 'num_date' uses node_attrs.num_date.value.",
    ] = "div"
    # ... and so on for every parameter
```

A single small helper in `cli.py` walks `PlotConfig.__dataclass_fields__`,
pulls each `Annotated` description out, and emits a click option with that
help text. The function's docstring is generated from the same source via
a one-line decorator that fills in the `Args:` block. mkdocstrings sees
the populated docstring and renders it; `mkdocs-click` sees the populated
click options and renders the CLI page. **The descriptions are written
exactly once, in the dataclass.**

### Rationale for the toolchain itself

- `mkdocstrings` introspects the `tap.plot` signature and `PlotConfig`
  dataclass and renders the function-signature docs from one source of
  truth.
- `mkdocs-click` renders the `tree-annotated-plot` CLI docs from the same
  click decorators that define the actual CLI. No separate manpage to
  maintain.
- MkDocs Material is one config file, builds in seconds, deploys to GitHub
  Pages with one workflow file.
- All three are actively maintained and lightweight.

(Sphinx + autodoc + sphinx-click would also work but is more configuration
for what we need. If the user already maintains other Sphinx docs, switch
on request.)

Files added:

- `docs/index.md` — landing page; project overview + 30-second example.
- `docs/python-api.md` — `::: tree_annotated_plot.plot` block;

example. Ordering reflects the dependency graph: chart loading + tip
reconciliation come early because the real-world H3N2/H1N1 charts can't be
exercised without them; the CLI lands once the function works end-to-end on
real data; the layout generalization (Phase 2g) comes after the H3N2
(strain-on-y) case works because H1N1 (strain-on-x) needs it.
